## Docker Compose для демонстрации маршрутизации, rate limiting и перенаправления на «простую версию»
### 1,2,3
Ниже приведён пример `docker-compose.yml`, который поднимает:
- **Kong** (API Gateway) с PostgreSQL в качестве базы данных конфигурации  
- **main-app** – основное приложение (сложная логика, например, работа с БД)  
- **simple-app** – простая версия (статический HTML или лёгкий API)  
- **redis** (опционально для распределённого rate limiting)

При превышении лимита запросов (например, 5 запросов в минуту) Kong автоматически возвращает HTTP‑статус `429 Too Many Requests`.  
Но в задании требуется **перенаправлять** на простую версию. Это можно сделать через **пользовательскую логику в Kong** – плагин `pre-function`, который перехватывает ответ 429 и превращает его в 302 Redirect.

Плагин rate-limiting в Kong по умолчанию реализует Fixed Window (интервал: minute = 60 секунд, окно обнуляется на границе минуты). Это самый простой алгоритм: за минуту разрешено 5 запросов с одного IP. При превышении — 429.

Алгоритмы Token Bucket или Sliding Window в Kong доступны через другой плагин: rate-limiting-advanced (требует Redis, позволяет выбрать window_type: sliding или strategy: redis-cluster с token bucket). В вашем примере — классический фиксированный лимитер без сглаживания.

### 4. Запуск и настройка


```bash
# Запускаем всё
docker compose down
docker compose up --build


# Дождаться готовности Kong (около 10–15 секунд)
# Можно проверить через curl http://localhost:8001/services
```

### 5. Демонстрация нагрузки и редиректа

**Нормальный запрос (в пределах лимита):**  
```bash
curl -i http://localhost:8000/api
```
Ответ – `200 OK`, содержимое `main-app/index.html`

**Быстро посылаем 6 запросов подряд:**  
```bash
for i in {1..6}; do curl -s http://localhost:8000/api | grep -o '<h1>.*</h1>'; done
```

- Первые 5 раз получим `<h1>СЛОЖНАЯ версия приложения</h1>`  
- На 6‑й раз – HTTP 307, браузер (или curl с `-L`) перейдёт на `/simple`, ответ – `<h1>УПРОЩЁННАЯ версия приложения</h1>`

Если использовать curl с авто-следованием редиректу:  
```bash
curl -L http://localhost:8000/api
```
После превышения лимита вы получите страницу simple-app.

### 6. Почему это работает?

- **Rate limiting** плагин Kong считает запросы по IP (при `policy: local` – в памяти контейнера). При превышении лимита он возвращает статус `429`.
- **pre-function** плагин вмешивается в жизненный цикл запроса. Мы переопределили функцию `kong.response.exit`. Если статус равен 429, мы возвращаем `307` с заголовком `Location`, ведущим на `/simple`.
- Маршрут `/simple` уже привязан к `simple-svc`, который отдаёт упрощённый контент.

### 7. Альтернативный подход без Lua (для демонстрации концепции)

Если не хотите использовать pre-function, можно настроить второй маршрут с более высоким приоритетом и проверять специальный заголовок, который проставляется при лимите – но это сложнее.  
Приведённый выше способ – самый наглядный для **обучения**: видно, как gateway может динамически менять поведение при превышении нагрузки.

## Итог: что иллюстрирует этот пример

| Компонент              | Что демонстрируется                                      |
|------------------------|----------------------------------------------------------|
| **Маршрутизация**      | `/api` → main-app, `/simple` → simple-app               |
| **Rate limiting**      | 5 запросов в минуту (настраивается через `minute: 5`)    |
| **Перенаправление при перегрузке** | 429 → 307 Redirect на простую версию               |
| **Service Discovery**  | В примере статические адреса, но можно заменить на Consul |
| **Docker Compose**     | Полная изолированная среда для экспериментов             |

Вы можете увеличить лимит в `kong.yml` (`minute: 100`, как в вашем примере) или добавить Redis для распределённого счётчика. Ключевая идея: **API Gateway может не только отказывать, но и gracefully деградировать** – перенаправлять на fallback‑сервис.
